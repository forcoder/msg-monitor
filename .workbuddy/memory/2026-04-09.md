## APK 编译成功

### 根因分析
项目因多个连锁问题无法编译：

1. **Kotlin 语法错误**（之前已修复）：`OtaModule.kt` 和 `OssModule.kt` 所有代码被注释但保留 import，导致 `Expecting a top level declaration`
2. **gradle.properties BOM 问题**：`gradle.properties` 文件开头有 UTF-8 BOM（EF BB BF），导致 Java Properties 解析器将第一行 key 解析为 `\ufefforg.gradle.jvmargs`，使 `org.gradle.jvmargs` 和 `org.gradle.java.home` 均失效，Gradle daemon 回退到 JDK 8 + 默认 512MB 堆，触发 OOM

### 修复措施
- 移除 `gradle.properties` 的 BOM：用 `[System.IO.File]::ReadAllBytes` 读取，跳过前3字节后重新写入（PowerShell 的 `WriteAllText` 默认会加 BOM，必须用字节方式处理）
- 复制 JDK 19 到无空格路径 `D:\jdk19`（原路径 `D:\Program Files\Java\jdk-19` 含空格，Windows 下 Java Properties 路径解析有问题）
- `gradlew.bat` 增加 `set JAVA_HOME=D:\jdk19` 和 `set GRADLE_OPTS=-Xmx4096m -Xms1024m -XX:MaxMetaspaceSize=512m -Dfile.encoding=UTF-8`
- `gradle.properties` 中 `org.gradle.jvmargs=-Xmx4096m -Xms1024m -XX:MaxMetaspaceSize=512m -Dfile.encoding=UTF-8`，同时增大堆到 4GB
- BOM 去除后 + 路径修复后，Gradle daemon 正确使用 JDK 19 + 4GB 堆，构建成功

### 构建结果
- APK 路径：`app\build\outputs\apk\debug\app-debug.apk`
- APK 大小：16.19 MB（初次编译）、16.20 MB（增量编译）
- 构建耗时：4m 20s（初次）、23s（增量）
- 编译参数：`.\gradlew.bat assembleDebug --no-daemon`

## 修复 "屡次停止运行" 运行时崩溃

### 根因
`KefuApplication` 没有 `@HiltAndroidApp`，但 4 个组件使用了 `@AndroidEntryPoint`：
- `MainActivity`、`NotificationListenerServiceImpl`、`FloatingWindowService`、`BootReceiver`
`@AndroidEntryPoint` 生成代码依赖 Application 有 `@HiltAndroidApp`（生成 `Hilt_XXX` 组件）。没有它，系统启动组件时抛出 `IllegalStateException`。
次要：`OtaUpdateWorker` 使用 `@HiltWorker` 但未配置 `HiltWorkerFactory`。

### 修复
- `KefuApplication.kt`：添加 `@HiltAndroidApp` + `Configuration.Provider`（注入 `HiltWorkerFactory`）
- `AppEntryPoint.kt`：更新注释

### 编译结果
- APK：`app\build\outputs\apk\debug\app-debug.apk`，16.2 MB
- 参数：`.\gradlew.bat clean assembleDebug --no-daemon`

## 修复消息监听和悬浮窗无响应（第二轮）

### 问题
应用安装后无法监听配置应用的消息，也不弹出悬浮窗。

### 根因（多层面）
1. **`monitoringEnabled` 默认为 false**：`PreferencesManager.UserPreferences` 中 `monitoringEnabled = false`，首启后即使授权了权限也不会自动开始监控
2. **权限引导顺序错误**：`MainActivity` 在 POST_NOTIFICATIONS 权限回调里同时调用 `requestOverlayPermission()` 和 `requestNotificationListenerPermissionIfNeeded()`，两者都调用 `startActivity()`，Android 只响应第一个，通知监听权限请求被忽略
3. **`FloatingWindowService` 崩溃**（之前修复）：`@AndroidEntryPoint` 组件依赖 `@HiltAndroidApp` 生成的 Hilt 组件，之前无 `@HiltAndroidApp` 时服务启动就崩溃

### 修复
1. `PreferencesManager.kt`：`monitoringEnabled` 默认值从 `false` 改为 `true`（首启自动启用监控）
2. `MainActivity.kt`：重构权限引导流程，使用 `pendingOverlayPermission` / `pendingNotificationPermission` 标志追踪等待状态，`onResume()` 检测权限到位后自动继续下一步，避免同时 `startActivity()` 被系统吞掉
3. `FloatingWindowService.kt`：在 `onCreate()` 加日志确认 Hilt 注入成功；Toast 消息改为引导用户返回 APP 顶部横幅

### 编译结果
- APK：`app\build\outputs\apk\debug\app-debug.apk`，16.2 MB
- 参数：`.\gradlew.bat clean assembleDebug --no-daemon`

## 修复消息监听/悬浮窗功能问题（第三轮）

### 问题
用户反馈功能不太正常，无法按预期监听消息并弹出窗口。

### 根因（5个问题）
1. **`initializeMatcher()` 造成无限 collect**：`KnowledgeBaseManager.initializeMatcher()` 使用 `Flow.collect()` 会永不终止，导致 matcherJob 永远处于活跃状态

4. **FloatingWindowService DisplayData null 时日志不详细**：调试信息不足
5. **iconObserverJob 生命周期管理**：每次 start() 时已正确 cancel 旧 job（无需额外修改）

### 修复
1. **KnowledgeBaseManager.kt**：`initializeMatcher()` 改用 `.first()` 一次性获取当前规则，不再持续 collect
2. **ReplyOrchestrator.kt**：`shouldSkipMessage()` 改用精确匹配替代 `contains`，避免误判真实消息
3. **PreferencesManager.kt**：`monitoringEnabled` 默认值确认已为 `true`（上轮已完成）
4. **FloatingWindowService.kt**：DisplayData null 时添加更详细的错误日志

## 修复悬浮窗崩溃问题（第四轮）

### 问题
用户反馈悬浮窗功能存在两个运行时崩溃：

### 崩溃 1：CalledFromWrongThreadException
- **日志**：`android.view.ViewRootImpl$CalledFromWrongThreadException: Only the original thread that created a view hierarchy can touch its views.`
- **位置**：`FloatingWindowService.performKnowledgeSearch` 方法
- **根因**：`serviceScope.launch` 默认在非 UI 线程执行，但 `removeAllViews()` 和 `addView()` 等 UI 操作必须在主线程
- **修复**：用 `withContext(Dispatchers.Main)` 包装 UI 操作，数据查询保持在 IO 线程

### 崩溃 2：View not attached to window manager
- **日志**：`java.lang.IllegalArgumentException: View=android.widget.FrameLayout{...} not attached to window manager`
- **位置**：`FloatingWindowService.showFloatingIconOnly` 第 268 行
- **根因**：代码顺序错误 — 先调用 `applyExpandedState()` 再调用 `windowManager.addView()`，但 `applyExpandedState` 内部会调用 `updateWindowLayout()`，此时 View 还未添加到 WindowManager
- **修复**：
  1. 调整代码顺序：先 `addView`，再 `updateFloatingContentForIconOnly()` 和 `applyExpandedState()`
  2. 在 `updateWindowLayout()` 添加防御性检查：`if (floatingView?.isAttachedToWindow != true) return`
