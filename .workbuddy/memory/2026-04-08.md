## 阿里云OSS自动上传系统实现

### 核心修复
1. **配置外置化**：将OSS域名、Bucket、AK/SK等敏感信息从代码中提取到配置文件
2. **自动上传集成**：编译完成后自动上传APK到OSS，支持自动版本升级

### 新增功能
1. **配置文件管理**：
   - `oss-config.properties`：主配置文件（包含敏感信息，不提交到Git）
   - `oss-config.properties.example`：配置示例文件（不含敏感信息）
   - 配置项：endpoint、bucket、access_key_id、access_key_secret等

2. **自动上传脚本**：
   - `upload-to-oss.ps1`：PowerShell自动上传脚本
   - 支持自动提取APK版本信息
   - 自动生成OSS签名和对象键
   - 自动计算文件MD5
   - 生成上传历史和版本信息文件

3. **构建脚本集成**：
   - `build-and-upload.bat`：编译+上传一键脚本
   - 更新`build.bat`：集成自动上传功能（根据配置决定是否上传）
   - 支持PowerShell版本检测（需要3.0+）

4. **代码重构**：
   - `OssConfig.kt`：新增配置管理类，从外部文件读取OSS配置
   - 修改`AliyunOssManager.kt`：移除硬编码配置，使用配置类
   - 支持配置验证、重载和动态获取

5. **版本自动升级**：
   - 上传成功后自动增加版本代码（versionCode+1）
   - 自动升级版本名称（如1.2.3→1.2.4）
   - 可配置是否启用自动升级

### 文件生成
- `upload-history.json`：上传历史记录（最近20次）
- `version-info.json`：版本信息文件（包含URL、MD5等）
- `.gitignore`：避免提交敏感配置文件和生成文件

### 使用流程
1. **首次配置**：
   ```bash
   copy oss-config.properties.example oss-config.properties
   notepad oss-config.properties
   ```

2. **编译上传**：
   ```bash
   build-and-upload.bat
   ```
   或直接上传现有APK：
   ```bash
   python upload_oss.py
   ```

3. **查看结果**：
   - 查看`upload-history.json`和`version-info.json`
   - 从生成的URL下载APK测试（需要Bucket设置为公共读或使用带签名URL）

### 实际测试结果

- 📍 **上传时间**：2026-04-08 21:44:05
- 📦 **文件信息**：15.97 MB，MD5: 715f4b3f8d64311aa30a504ccfa94016

- ⚠️ **访问限制**：Bucket默认私有，需要设置公共读或使用带签名URL

### 技术要点
1. **时间同步**：必须使用UTC时间，服务器时间差超过15分钟会导致`RequestTimeTooSkewed`错误
2. **Content-MD5**：必须是文件内容MD5的base64编码，而不是MD5字符串的base64编码
3. **URL构建**：`endpoint`已包含bucket前缀，URL应为`https://{endpoint}/{object_key}`
4. **认证签名**：使用HMAC-SHA1算法生成签名，包含方法、MD5、类型、时间和资源路径

## 其他修复（原有内容保留）

- 修复 Excel 导入"成功 3 条，失败 178 条"的 bug：根本原因是 `normalizeCsvHeader` 把中文表头 lowercase 后，`isCsvHeaderRow` 里的 knownHeaders 没有对应中文列名，导致整行表头被当作数据行，走了旧版固定列模式全部解析失败。修复方案：①`normalizeCsvHeader` 对含非 ASCII 字符的列名保留原样；②`isCsvHeaderRow` 补充所有中文列名（触发条件、触发类型、回复内容、适用房源、状态、规则名称等）；③`parseCsvRule` 对触发条件/触发类型/状态等中文列名直接读取，并在关键词为空时把触发条件原文当关键词使用；模拟测试百居易 v3.xlsx（173 行）成功率从 3/173 提升到 173/173，并通过 `gradlew.bat assembleDebug` 验证构建成功。

- 已为消息监听添加占位消息过滤：在 `ReplyOrchestrator.handleNewMessage` 中新增 `shouldSkipMessage` 方法，排除"给你发送了新消息"、"发送了新消息"、"向你发送了一条消息"、"新消息来了"等占位通知，减少无用悬浮窗触发。同时过滤太短内容（<3字）和 [图片]/[表情]/[文件] 等媒体占位文本。

- 已在悬浮窗添加"知识库"标签页：在 `FloatingWindowService` 的展开面板中新增"建议回复"与"知识库搜索"双标签页切换。知识库标签页包含搜索框和结果展示区，支持关键词搜索知识库规则，点击"复制"按钮可一键复制话术到剪贴板。相关改动已通过 `gradlew.bat assembleDebug` 验证。

- **修复知识库显示问题**：
  1. **规则列表显示优化**：移除规则卡片中显示的 `[CONTAINS] 通用问题` 等冗余信息。现在只在分类非空且不是"通用问题"时才显示分类，移除了匹配类型（CONTAINS/EXACT/REGEX）的显示，让界面更简洁。
  2. **编辑对话框优化**：将分类字段默认值从"通用问题"改为空字符串，标签改为"分类（可选）"，添加占位符说明"如：入住咨询、退房问题、价格咨询等"。保存时不再自动填充"通用问题"作为默认值。
  修改文件：`app/src/main/java/com/csbaby/kefu/presentation/screens/knowledge/KnowledgeScreen.kt`

- **修复监控app新消息自动回复不弹出的问题**：
  问题原因：`shouldSkipMessage` 方法过滤条件过严，将长度小于3个字的所有消息都过滤掉，导致简短真实回复（如"好的"、"行"等）被误过滤。
  修复方案：放宽过滤条件：
  1. 允许常见的简短回复通过（"好"、"行"、"OK"、"收到"、"嗯"等）
  2. 只有空内容才直接过滤
  3. 长度小于2个字且不是常见简短回复才过滤
  修改文件：`app/src/main/java/com/csbaby/kefu/infrastructure/reply/ReplyOrchestrator.kt` 第324-336行

- **修复应用崩溃问题："屡次停止运行"**：
  问题分析：应用可能在多个点崩溃，包括：
  1. `KefuApplication.onCreate()` 中 `replyOrchestrator.start()` 可能因依赖注入问题导致空指针
  2. `ReplyOrchestrator` 内部处理消息时可能抛出未捕获异常
  3. 悬浮窗服务可能因权限或视图问题崩溃
  
  修复方案：
  1. **应用层保护**：在 `KefuApplication.kt` 中添加try-catch和空检查
  2. **服务层保护**：在 `ReplyOrchestrator.kt` 的 `start()` 和 `handleNewMessage()` 方法中添加异常处理
  3. **恢复简单过滤逻辑**：暂时恢复简单的消息过滤（长度<2就过滤），避免复杂逻辑引入新问题
  
  修改文件：
  - `app/src/main/java/com/csbaby/kefu/KefuApplication.kt`：添加安全启动逻辑
  - `app/src/main/java/com/csbaby/kefu/infrastructure/reply/ReplyOrchestrator.kt`：多层异常保护
